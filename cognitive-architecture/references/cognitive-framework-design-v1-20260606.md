# 三轴坐标 + 因果链认知框架设计文档
> 版本: v1.0-draft | 日期: 2026-06-06 | 作者: Hermes (受Joe引导articulate)

---

## 0. 核心理念

**参数不是知识，坐标才是。**
当前 LLM 将知识编码为连续参数浮点数——不可索引、不可校验、不可单独更新。
本框架将知识从参数中剥离为可导航的坐标空间，保留神经网络做两件事：**导航决策**和**简短生成**。

**检索即思考。** 在坐标空间中沿因果链的每一次移动，本身就是一次推理步骤。
路径选择 = 推理过程。路径终点 = 回答依据。

## 1. 三轴坐标系统

每个知识节点在三维连续坐标空间中拥有唯一地址。

### 1.1 领域轴（X轴 — 横向）

> 位置本身不带权重。横向的关联强度只存在于边里。

- 不同领域/事实按极角分布在横轴上
- 位置仅用于导航——知道一个节点在哪 = 知道如何找到它和附近节点
- **横向边（领域间关联）有方向权重：A→B 和 B→A 粗细分开记录**

#### X轴坐标分配机制

新知识进入时，沿因果链往下溯源，找到最接近的已有节点：

1. 如果能锚定到已有节点 → 挂在该节点附近，用边表示关系
2. 如果找不到能锚定的位置 → **进入候选隔离区**（见 4.1）
3. 候选区知识等待：更多知识的交叉验证 或 新现象的支持
4. 验证通过 → 挂入主坐标空间，边权重从薄开始自然生长

**（存疑①）坐标分配的决策者**
新知识的坐标分配依赖系统自身的判断能力。如果系统的已有坐标覆盖不到该领域，新知识可能被错误定位到相邻位置，污染因果结构。
→ Joe的回答：依靠验证机制——不能解释现象或不与已有知识联系的新知识被隔离开，等待其他知识或现象验证。

### 1.2 时间轴（Y轴 — 纵向）

- 新的一般大于旧的（时间向量单向递增）
- **影响遗忘机制**：时间越久远的节点，如未被访问，优先被压缩/归档/遗忘
- 时间轴不是"删除"，是"降权"

#### 遗忘策略（待细化）

```
高频访问的节点 → 时间影响小 / 权重保持
低频访问且久远的节点 → 边权重衰减 → 节点标记为可归档
归档节点 → 从主导航空间移除，存入冷存储
```

**（存疑②）归档后的节点能否恢复？**
如果后续路径发现需要已归档节点，系统是否具备反归档能力？
→ 当前设计没有明确回答。一种可能：导航走到一处"断裂"（相邻坐标没有节点），触发冷存储检索。

### 1.3 密度轴（Z轴 — 颗粒度）

密度轴是知识抽象程度的刻度。从具体事实到抽象原则分 3-4 层：

| 层 | 名称 | 含义 | 示例 |
|---|---|---|---|
| 0.01 | 事实 (Fact) | 不可再分的基础观察 | "API回403"、"服务器IP是192.168.1.1" |
| 0.1 | 推导经验 (Derived) | 从事实归纳的操作性知识 | "403时先检查token是否过期" |
| 0.5 | 表现 (Manifestation) | 模式级别的规律 | "认证服务超时会导致连锁401" |
| 1.0 | 抽象结论 (Abstract) | 跨领域适用的原则 | "系统的可靠性取决于其最弱认证环节" |

#### 密度轴的工作流程

1. 查询进入 → 系统先判断查询的"期望密度"
2. 导航到对应密度层的节点范围
3. 结果不够深入 → 沿密度轴下钻一层（走向更具体的事实）
4. 结果太琐碎 → 沿密度轴上浮一层（走向更抽象的模式）
5. 更多关联问题 → 沿因果链扩展

## 2. 因果链（边系统）

### 2.1 边类型

四种基础边类型，继承自之前的因果链设计：

| 类型 | 含义 | 方向 |
|---|---|---|
| `depends-on` | A依赖B，B是A的前提 | 有向 |
| `uses` | A使用B作为工具/输入 | 有向 |
| `avoids` | A绕开B，B是A的替代方案 | 有向 |
| `related` | A和B在领域轴相邻但无直接推导关系 | 双向（可非对称加权） |

### 2.2 边权重

每条边有**两个独立权重**：A→B 和 B→A 可以粗细不同。

权重的影响因素：
- **推导确定性**：因果关系的强度（如"稳定复现"比"偶尔触发"的边更粗）
- **使用频率**：路径被导航穿过的次数（**情境权重覆盖**见 2.3）
- **时间衰减**：未被访问的边随时间变细

### 2.3 边粗细的双因子决定

边的粗细**不是单一指标**，而是两个独立因素共同作用的结果：

- **相关性（Correlation）**：节点之间的固有关系强度，由节点本身的性质决定，变化慢。属于静态权重。
- **热度（Heat）**：由两个子项构成——
  - **频次热度**：该边被路径选中的调度次数（长期累积，跨查询共享）
  - **临时热度**：当前单次查询独有的临时修正，仅在该次查询的生命周期内有效

**每次查询有自己独立的临时热度层。** 同一张图，同一组相关性权重，但每个查询在临时热层里做自己的路径选择。查询结束后，临时层散去，频次计数器+1。

一次查询的热量场与其他查询完全隔离，不存在"热量场打架"。

边粗度 = 相关性 × (1 + α·频次热度 + β·临时热度)
_（α 和 β 为各自的缩放系数，具体值待实测确定）_

### 2.4 自修正机制

> Joe: "发现解释不了就知道回头了"

**"感觉不对"的机制化：**
系统沿一条路径导航时，如果在某条边上遇到预期与实际不符的情况（如一条本应很粗的边在当前上下文中出奇地细），会触发一个"检查"信号。

检查流程：
1. 暂停当前路径
2. 回溯一步，检查当前节点的其他候选边
3. 如果候选边中有更符合语境的 → 切换路径
4. 如果所有候选边都不符合 → 触发"悬停"（等待更多上下文）

这个机制不需要额外的元认知模块——**图拓扑本身的特性触发自我修正**。路径走到了桥突然变细的地方，自然会换一条。

## 3. 导航 = 推理

### 3.1 标准查询流程

```
1. 接收查询
2. 估算查询在密度轴上的预期深度（表层问答 / 经验推导 / 深度分析）
3. 沿领域轴定位到对应区域
4. 检索该区域的节点（优先当前密度层）
5. 如果节点信息不足：
   a. 需要更细 → 沿密度轴下钻（fact方向）
   b. 需要更宽 → 沿因果链扩展（follow edges）
   c. 需要更高视角 → 沿密度轴上浮（abstract方向）
6. 收集够信息 → 使用小模型在叶子节点生成回答
7. 回答中包含路径来源
```

### 3.2 与向量检索的区别

| | 向量检索 | 坐标导航 |
|---|---|---|
| 匹配方式 | 语义相似度 | 坐标 + 路径 |
| 结果性质 | 最像的top K | 路径上的节点集合 |
| 可解释性 | 黑箱 | 路径可回溯 |
| 边界案例 | embedding不够好就偏 | 坐标不够精确也偏，但能知道偏向了哪 |
| 多跳推理 | 需要额外RAG管道 | 边本身就支持多跳 |

### 3.3 典型路径长度

**（存疑④）路径长度的性能影响**

一个查询走 5 跳因果链，每跳做一次"下一步去哪"的决策。
即使每次决策 500ms，2.5s 过去了，还不包括从叶子生成回答的时间。
和直接问大模型比——3-5s内给回答——这个架构在时间上可能不占优。

→ Joe的回答："时间随便 反正是探索"
→ 当前阶段：探索用，性能不是瓶颈。但后续生产化时需要关注。

## 4. 知识生命周期

### 4.1 节点状态标签

所有节点必须有状态标签，**坐标体系不分离**：

| 真值 | 含义 | 边使用 | 回答行为 |
|---|---|---|---|---|
| `true` | 经路径验证、交叉确认 | 权重稳定，可作为可靠前提 | 给出置信回答 |
| `unknown` | 存在但真值不确定——**合法态，不是过渡态**。新知识默认进入此态 | 边保留但带不确定性标记，路径可穿过，但不作为可靠证据链 | 回答标注"不确定"，可附带来源供判断 |
| `false` | 外部确认该节点内容错误 | 出边断掉，下游节点继承 unknown 而非直接崩塌 | 不进入回答路径，除非被显式引用为"错误案例" |
| `hypothesis` | 推导性节点，未充分验证 | 边带漂移余量 | 标注"推测" |
| `raw` | 候选隔离区中的观察 | 无有效边 | 不进入回答路径 |
| `isolated` | 知道坐标但无任何连线 | 无连线 | 不参与导航 |
| `archived` | 从主图移入冷存储 | 无权重 | 不可直接使用 |

同一查询可以穿过不同真值区：沿 `true` 走主干道，到边界附近从 `unknown` 或 `hypothesis` 区借路，走不通回头。

### 4.2 新知识插入

```
新知识进入系统
  ├→ 沿因果链往下溯源
  │   ├→ 找到锚定节点 → 计算坐标 → 搭建边 → 标记为 hypothesis
  │   └→ 找不到锚定 → 放入 raw 区（候选隔离区）
  └→ 等待验证：
      ├→ 后续知识经过并锚定到它 → 验证强度增加
      ├→ 多个现象指向同一个 raw 节点 → 升级为 hypothesis
      └→ 长期无访问 → 自动归档
```

### 4.3 升级机制：hypothesis → verified

升级标准不再需要独立的裁判模块。

**节点被路径选中的频率计数器 = 置信度。**
当一个 hypothesis 节点的频率计数器增长趋势持续超过阈值时——即被越来越多的独立路径穿过——系统将其标记为 `verified`。

```
upgrade_condition = count(hypothesis_node命中) / count(总路径) > threshold
```

升级动作本身不需要额外推理——计数器到达阈值就是升级信号。升级后该节点的相关性权重可被后续路径视为稳定值。

**（存疑⑤）阈值设定**
"频率增长到多少算 verified"——这个阈值是固定值还是动态的？
→ 待实测确定。初始建议：固定阈值，观察后迭代。

---

### 4.4 真值变化：true → unknown → false

当一个 `true` 节点被外部信号或交叉检验发现可能有问题时，系统将其标记为 `unknown`。

**unknown 的规则：**
- 边保留，但带"不确定性标记"
- 路径可以穿过，但不视为可靠证据
- 依赖 `unknown` 节点的下游节点**不崩塌**——它们的真值不变，但系统在回答时标注"依赖不确定节点"
- unknown 是**合法态**，不是待处理队列。节点可以无限期处于 unknown 状态

**false 的传播不引起级联崩塌：**
标记为 `false` 时：
1. 该节点的出边断掉
2. 下游节点**不降级**——仅其边权重因失去一条稳定前提而被调低，真值不变
3. 后续路径穿到下游节点时，系统发现"这个节点的一条重要前提已断，路径可靠度降低"

**三进制真值传播规则：**
```
true  + true    → true
true  + unknown → unknown
unknown + unknown → unknown
false + 任意    → 仅降低关联路径权重，不传播 false
```

**（存疑②）假确认的标准**
suspected → false 的升级靠什么触发？外部手动确认？多条独立路径交叉检验？还是某种容错阈值？
→ 当前未定义。猜测：需要外部信号（人为标记）或足够的矛盾路径累计。纯自动化判断风险高。

## 5. 存储与索引

### 5.1 物理存储

```
主空间（热存储）：
  - 节点表：id, x, y, z, content, status, created_at, updated_at
  - 边表：source_id, target_id, type, weight_fwd, weight_bwd, relationship_strength
  - 路径日志：query_id, node_sequence, final_outcome

冷存储：
  - 归档节点：内容不变，移除自导航索引
  - 归档触发条件：n 轮时间单位无访问 + 边权重低于阈值
```

### 5.2 索引方式

- **主索引**：三维坐标 R-tree（范围查询："x0-x1 ∩ y0-y1 ∩ z0-z1"）
- **辅助**：因果链邻接表（给定节点查所有相连节点）
- **备用**：FTS5 文本搜索（仅用于"坐标不明确时兜底"）

**（存疑⑥）R-tree 在因果链导航中的适用性**
R-tree 适合空间范围查询（"找这个坐标附近的节点"），但因果链导航需要的是图遍历（"从节点A出发，沿着边到节点B的邻域"）。
两个查询模式需要两套索引——一个按坐标，一个按邻接。这在小规模（<10万节点）可能不是问题，但需要明确的设计选择。

## 6. 与其他系统的关系

### 6.1 Hermes 现有系统

| 组件 | 在本框架中的角色 | 兼容性 |
|---|---|---|
| memory-db (SQLite + FTS5) | 存储层？或作为冷存储 | (存疑) 需要改造以支持坐标索引 |
| 4K memory | 暂存"当前问询的路径热量" | 兼容，路径日志可直接写入 |
| embedding service (8082) | 坐标定位的辅助工具 | 不是核心——坐标本身就是地址 |
| 小模型 (2B-4B) | 导航决策者 + 叶子节点回答生成 | 是核心，但角色被压缩 |



### 6.2 当前不做

- **不做分布式**：单机单进程，所有节点在内存或本地 SQLite
- **不做实时同步**：知识更新通过写入操作完成，不需要实时广播
- **不做多用户隔离**：当前设计假设单一用户/单一知识体

## 7. "（存疑）" 汇总

| 编号 | 存疑 | 状态 |
|---|---|---|
| ① | 坐标分配的决策者——系统自身判断能力不足时是否会污染因果结构 | ✅ 有方案：验证机制+隔离区 |
| ② | true→unknown→false 的触发条件——什么信号触发 true→unknown，什么触发 unknown→false | ❌ 当前依赖外部信号，纯自动化风险高 |
| ③ | 多条查询的热量场叠加冲突 | ✅ 已解决：每次查询独立临时热度层，不受其他查询干扰 |
| ④ | 路径长度的性能影响——典型路径是否比直接问大模型慢 | ✅ 接受，当前阶段不是瓶颈 |
| ⑤ | hypothesis→verified 升级阈值——固定值还是动态 | ⚠️ 方向明确（频率计数器），具体系数待测 |
| ⑥ | R-tree 在因果链导航中的适用性——两种查询模式需要两套索引 | ⚠️ 小规模不痛，路线图里需注明 |
