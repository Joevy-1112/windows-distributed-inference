## 一、"检索即思考"漏斗（对 v1 §3 导航机制的深化）

v1 的导航是纯坐标 + 因果链，缺少"初始定位"阶段。在坐标完备之前，需要一个混合漏斗做入口：

```
语义召回（嵌入 top-100）
  → 精确过滤（SQL/FTS5 条件筛选）
  → 注意力排序（importance × recency × context_match）
```

v1 说"向量不是核心——坐标本身就是地址"。这对完备坐标是成立的，但当前坐标不完整。**漏斗是坐标完备前的过渡层，坐标完备后漏斗退化为入口层。**

## 二、因果连线双维度权重（对 v1 §2.2-2.3 的细化）

v1 定义了边粗细 = 相关性 × 热度，但**没有区分因和果**。

本日 Joe 提出的新维度：

| 权重 | 含义 | 作用 |
|:-----|:-----|:-----|
| **因权重** (Causal Strength) | "A 发生导致 B 发生的概率/强度" | 决定"该不该走这条路" |
| **果权重** (Consequence Importance) | "B 发生后对系统状态的影响程度" | 决定"走到终点后值不值得停" |

两者分开建模才能区分：
- 强因果弱影响：代码格式化触发 Lint 警告（因 0.9，果 0.1）
- 弱因果强影响：闲聊中提到喜欢咖啡（因 0.2，果 0.8）

### 与 v1 边权重模型的融合

v1 边权重：`edge_weight = correlation × (1 + α·freq_heat + β·temp_heat)`

新模型：每条边有两个向量：
```
edge = {
  source → target: {
    cause_weight:  0.7,   // 因果强度（替代原 correlation 的"推导确定性"）
    effect_weight: 0.4,   // 果影响力（新维度，影响停留决策）
    freq_heat:     12,    // 频次热度（v1 原样保留）
    temp_heat:     0.8    // 临时热度（v1 原样保留，per-query 隔离）
  }
}
```

**导航时的决策：**
- 路径选择（"下一步走哪"）→ 看 `cause_weight × heat`
- 停留决策（"这个节点够不够"）→ 看 `effect_weight`

v1 的"相关性"被拆成两个更精确的维度。

## 三、压缩机制：不可逆 → 可追溯（对 v1 §4 知识生命周期的补充）

v1 的 §4 只覆盖了节点状态流转（hypothesis → true → unknown → false → archived），**没有覆盖"多个原始事件如何压缩成一个高层规则"**。

### 核心思想

压缩不是"替代"原始对话，是**生成一层索引**：

```
原始对话（state.db，永久保留）
  ├─ 压缩触发
  └─ 压缩摘要节点（高层规则）
       derived_from → 原始节点（因果连线，权重 ≈ 1.0）
       ↓
     追溯时沿 derived_from 回到原始对话
```

### 新节点状态和字段

新增字段（加入节点表）：
- `derived_from` — 压缩来源节点 ID 列表（JSON）
- `compressed_to` — 压缩目标节点 ID

新增边类型：
- `derived-from` — 高层节点派生自低层节点（因果权重 ≈ 1.0，语义等价）

### 为什么 1M 上下文也需要压缩

1. 跨会话长期积累超过 1M
2. 图节点膨胀导致检索开销线性上升
3. 长期重复模式压成一条高层节点 → 降低遍历深度
4. 压缩不是忘记，是可折叠可展开的细节

## 四、认知总管：小模型作为系统级调度器（全新组件）

v1 没有"调度层"概念——所有导航决策都在查询时即时完成。

本日设计：**一个 1-3B 小模型常驻显存，担任三个职能：**

| 职能 | 描述 |
|:-----|:-----|
| **语义路由** | 判断 query 走本地 14B / DSv4 / 直接回答 / 需要检索 |
| **实时监听** | 每轮对话异步判断是否触发压缩 |
| **压缩写入** | 生成摘要节点 + 挂因果连线（derived-from） |

### 与 v1 的关系

v1 的导航决策者是"小模型做导航决策"（§6.1），但范围限定在**知识图内导航**。总管模型把这个角色扩展为**系统级调度**——不仅决定"怎么走图"，还决定"走不走图"、"什么时候压缩"、"什么时候记账"。

### 部署方案

- 模型：1-3B（推荐 ModernBERT 或 Arch-Router 1.5B，或 embedding 模型 + 分类器快速验证）
- 常驻显存 2-4GB，部署在桌面 RX6600
- 异步处理，不阻塞主对话
- 自身产生的日志不触发新压缩

## 五、4K 宪法层 vs memory-db 图书馆层（新架构澄清）

v1 的 §6.1 列了 Hermes 现有组件的关系，但没有明确分层。本日澄清：

```
┌──────────────────────────────┐
│ 4K 宪法层（原生，手动维护）    │
│ MEMORY.md + USER.md           │
│ → 全量注入 system prompt     │
│ → 不自动写入，不自动修改      │
└──────────────────────────────┘
┌──────────────────────────────┐
│ memory-db 图书馆层            │
│ 结构化事实 + 漏斗检索         │
│ → 模型调 tool 按需检索       │
└──────────────────────────────┘
┌──────────────────────────────┐
│ 认知系统（独立）              │
│ 因果推理图 + 动态路由 + 总管  │
│ → 将来无缝挂接到 memory-db   │
└──────────────────────────────┘
```

这不是 RAG。RAG 是每轮自动 embedding → 注入 prompt。这是 Tool-based Long-term Memory——模型决定何时检索。

## 六、新 memory-db Schema（含认知预留字段）

```sql
CREATE TABLE memories (
    id              INTEGER PRIMARY KEY,
    content         TEXT,
    category        TEXT,
    tags            TEXT,        -- JSON array
    importance      REAL,
    source_session  TEXT,        -- 引线 → session_search
    message_range   TEXT,        -- "msg_42-msg_87"

    -- 注意力漏斗
    access_count    INTEGER,
    last_accessed   TEXT,
    decay_score     REAL,

    -- 嵌入
    embedding       BLOB,        -- 384维 float32

    -- 认知系统预留（不用就是 NULL，将来无缝挂接）
    causal_in       TEXT,        -- JSON: [{"node":3,"cause_w":0.7,"effect_w":0.4}]
    causal_out      TEXT,        -- JSON: [{"node":7,"cause_w":0.5,"effect_w":0.9}]
    derived_from    TEXT,        -- 压缩来源节点 ID
    compressed_to   TEXT,        -- 压缩目标节点 ID
    layer           INT          -- 图层级，方便 BFS 剪枝
);
```

## 七、与 v1 的差异总结

| 领域 | v1 (2026-06-06) | 新增 (2026-06-30) |
|:-----|:----------------|:-------------------|
| 边权重 | 相关性 × 热度（单维度） | 因权重 + 果权重（双维度） |
| 导航入口 | 纯坐标定位 | 漏斗（语义→过滤→注意力）过渡到坐标 |
| 压缩 | 未覆盖 | 可追溯压缩（derived-from 边） |
| 调度 | 无 | 认知总管小模型（路由+监听+压缩） |
| 存储层 | 知识图独立 | 认知字段预留在 memory-db 中，无缝衔接 |
| 4K 定位 | "暂存当前热量" | 宪法层——铁律/环境/画像，始终在线 |
